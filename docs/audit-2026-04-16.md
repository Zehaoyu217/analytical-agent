### P0-1: Health endpoint path mismatch — deploy will fail

**Location:** `backend/app/api/health.py`, `infra/docker/docker-compose.yml:49`, `infra/helm/claude-code/values.yaml:127`

**What's happening:**  
The backend exposes exactly one health route: `GET /api/health`. But:
- `docker-compose.yml` health check calls `http://localhost:3000/health/live`
- Helm liveness probe calls `/health/live`
- Helm readiness probe calls `/health/ready`

None of these paths exist. Docker Compose will loop-restart the container indefinitely. Kubernetes will never mark the pod as ready — the deployment never completes.

**Fix:**  
Either add the missing routes to the backend:
```python
@router.get("/health/live")
async def liveness() -> dict: return {"status": "ok"}

@router.get("/health/ready")
async def readiness() -> dict:
    # optionally check DB connection
    return {"status": "ok"}
```
Or update the infra config to call `/api/health`. Recommend the former — `/health/*` is the standard convention and keeps the API namespace clean.

### P0-2: `_lookup_frame` raises `NotImplementedError` — three tools will crash

**Location:** `backend/app/harness/skill_tools.py:43-48`, called at lines 197, 213, 306

**What's happening:**  
Three production tools — `correlate`, `compare`, `profile` — attempt to load a DataFrame like this:
```python
df = pd.DataFrame(args["data"]) if "data" in args else _lookup_frame(args)
```
`_lookup_frame` unconditionally raises `NotImplementedError`. This is intentional: a comment says "wire DuckDB lookup in higher layer." But no higher layer has wired it. If the LLM agent calls any of these tools without including a `data` key in the arguments — which is the likely path when the agent wants to reference a previously-loaded dataset — the tool crashes with an unhandled exception, breaking the agent turn.

**Fix:**  
Implement DuckDB-backed frame lookup. Pattern: look up the frame name in `ArtifactStore`, retrieve the stored parquet/JSON, return as DataFrame. Alternatively gate the three tools behind a UI-level guard that forces `data` to be passed inline, and document the constraint clearly in the tool schema so the model always includes it.

### P0-3: CORS hardcoded to `localhost:5173` — backend unreachable from any other origin

**Location:** `backend/app/main.py:54`

```python
allow_origins=["http://localhost:5173"],
```

**What's happening:**  
Any deployment that isn't local dev will have the frontend at a real domain. Every browser request will be blocked by CORS. This includes Docker Compose prod, Kubernetes, and any staging environment.

**Fix:**  
Read from environment:
```python
cors_origins = os.getenv("CORS_ORIGINS", "http://localhost:5173").split(",")
CORSMiddleware(allow_origins=cors_origins, ...)
```
Add `CORS_ORIGINS` to `.env.example` and Helm `values.yaml`.

### P0-4: Grafana dashboards reference metrics that don't exist

**Location:** `infra/grafana/dashboards/overview.json`, `infra/grafana/dashboards/conversations.json`

**What's happening:**  
The dashboards query Prometheus metrics including `http_requests_total` and `active_sessions`. The backend has zero Prometheus instrumentation — no `prometheus-fastapi-instrumentator`, no custom metric exports, no `/metrics` endpoint. The dashboards will show empty panels on every chart. This makes observability misleading: the system appears instrumented but is not.

**Fix (two options):**
1. Add instrumentation: `pip install prometheus-fastapi-instrumentator`, register at app startup, export `/metrics`. Define `active_sessions` as a custom gauge updated by session lifecycle events.
2. If observability is out of scope for now: remove or stub the Grafana dashboards so they don't imply false coverage.

### P1-1: `SESSION_SECRET` auto-generated per Helm deploy — invalidates all sessions on upgrade

**Location:** `infra/helm/claude-code/templates/secret.yaml:15`

```yaml
SESSION_SECRET: {{ default (randAlphaNum 64) .Values.secrets.sessionSecret | quote }}
```

**What's happening:**  
If `secrets.sessionSecret` is not set in `values.yaml`, Helm generates a new random 64-char string on every `helm upgrade`. Every running session is immediately invalidated. Users get logged out silently. This will happen on every deploy unless the operator knows to pin the value.

**Fix:**  
Remove the `randAlphaNum` default. Make it fail loudly if not provided:
```yaml
{{- if not .Values.secrets.sessionSecret }}
  {{- fail "secrets.sessionSecret must be set — no auto-generation" }}
{{- end }}
SESSION_SECRET: {{ .Values.secrets.sessionSecret | quote }}
```
Document this as a required pre-deploy step.

**Location:** `backend/app/api/slash_api.py:52-56`

@router.post("/execute")

    if payload.command_id not in _KNOWN_IDS:

The frontend's `SlashMenu` component sends these requests and presumably shows the success. Users see feedback but nothing happens. ### P1-3: No error boundaries in React — any component crash kills the whole UI

**Location:** Frontend-wide, verified by grep

**What's happening:**  
There are no `ErrorBoundary` components anywhere in the React tree. A runtime error in `VegaChart` (malformed Vega spec), `MermaidDiagram` (bad diagram syntax), or `MarkdownContent` (edge-case input) will propagate uncaught and unmount the entire application. The user sees a blank screen with no recovery path.

**Fix:**  
Add error boundaries at three levels:
1. `App.tsx` — catch-all for catastrophic failures
2. `SessionLayout.tsx` — isolate chat from other sections  
3. `RightPanel.tsx` — isolate artifact/tool panels from the chat

```tsx
<ErrorBoundary fallback={<PanelError message="Chart failed to render" />}>
  <VegaChart spec={spec} />
</ErrorBoundary>
```

### P1-5: Artifact name lookup has case mismatch bug — lookup always fails

**Location:** `backend/app/artifacts/store.py:226`

```python
nlower = name.lower()
for a in self._cache.get(session_id, []):
    if a.name == nlower or a.title.lower() == nlower:  # BUG: a.name is not lowercased
```

**What's happening:**  
`a.name` is stored as the original slug (e.g., `"Correlation_Matrix"`). `nlower` is the lowercased lookup key (`"correlation_matrix"`). The equality check `a.name == nlower` will almost always fail. Only the `a.title.lower() == nlower` branch saves it. Any code path that uses name-based lookup (vs. title) silently misses the artifact.

**Fix:**  
Normalize `a.name` to lowercase at storage time, or lowercase it in the comparison:
```python
if a.name.lower() == nlower or a.title.lower() == nlower:
```

### P1-6: Chart marker injection — user code can spoof artifact boundaries

**Location:** `backend/app/api/chat_api.py:82-100`

**What's happening:**  
The sandbox stdout parser uses sentinel strings (`__VEGA_SPEC__`, `__SAVED_ARTIFACT__`) as line markers to detect chart output. If user-executed code deliberately or accidentally prints these strings, the parser will misinterpret the output as an artifact boundary. This could corrupt the artifact store or inject unexpected chart data into the session.

**Fix:**  
Replace plaintext sentinels with unpredictable per-turn UUIDs generated at sandbox boot:
```python
CHART_SENTINEL = f"__VEGA_{uuid4().hex}__"
```
The sentinel is injected into the sandbox bootstrap and used for that turn only. User code cannot predict it.

### P1-11: No Helm RBAC — pods fail on clusters with RBAC enforced

**Location:** `infra/helm/claude-code/templates/` — no `role.yaml` or `rolebinding.yaml`

**What's happening:**  
A `ServiceAccount` is created by the chart but no `Role` or `RoleBinding` grants it any permissions. On clusters with RBAC enabled (which is the default on all managed Kubernetes services: GKE, EKS, AKS), the pod will fail to start if it needs any API access. If it doesn't need API access right now, the missing RBAC is a latent issue that will surface when any future feature needs to interact with the Kubernetes API.

**Fix:**  
Add minimal RBAC with least-privilege. At minimum, `templates/role.yaml` and `templates/rolebinding.yaml` binding the service account to an empty role (to make RBAC intent explicit and extensible).

### P1-12: No NetworkPolicy — any pod can talk to the agent backend

**Location:** `infra/helm/claude-code/templates/` — no `networkpolicy.yaml`

**What's happening:**  
Without a `NetworkPolicy`, any pod in the cluster can send requests to the agent backend. This matters because the backend executes arbitrary Python code in sandboxes and handles LLM output. Lateral movement from a compromised pod is unconstrained.

**Fix:**  
Add a `NetworkPolicy` that allows ingress only from the ingress controller and from same-namespace pods with a specific label selector. Deny all other ingress.

### P1-14: `as any` type cast in VegaChart — hides runtime type errors

**Location:** `frontend/src/components/chat/VegaChart.tsx`

```typescript

### P2-1: Silent exception handlers — 7 locations swallow errors invisibly

**Locations:**

| File | Line | What's swallowed |
|------|------|-----------------|
| `app/artifacts/events.py:34-35` | EventBus.emit | All subscriber exceptions |
| `app/scheduler/engine.py:87-98` | remove/pause/resume_job | APScheduler errors |
| `app/data/db_init.py:96-98` | `_try_date_range()` | Date inspection failures |
| `app/api/chat_api.py:98-99` | Chart capture | Chart parse errors |
| `app/api/chat_api.py:528` | Artifact JSON parse | Malformed artifact JSON |
| `app/api/chat_api.py:964` | `HookRunner.run_stop()` | Post-turn hook failures |
| `app/api/models_api.py:95-96` | Ollama capability check | Model probe failures |

**What's happening:**  
In each case, an exception is caught and execution continues without logging the failure. Debugging becomes extremely difficult: the system "works" but silently degrades. Hook failures are particularly bad — a broken post-turn hook goes completely unnoticed.

**Fix:**  
Minimum: add `logger.warning("...", exc_info=True)` in every `except Exception: pass` block. Better: only swallow errors that are truly non-critical (e.g., EventBus subscriber failures where one bad subscriber shouldn't break others). For hook failures, at minimum emit an SSE warning event so the frontend can surface it.

### P2-7: No `startupProbe` in Kubernetes — pod killed during slow startup

**Location:** `infra/helm/claude-code/templates/deployment.yaml`

**What's happening:**  
The liveness probe starts checking immediately. If the FastAPI app takes more than `initialDelaySeconds` to start (due to DuckDB initialization, skill registry loading, schema setup), the probe fails and Kubernetes kills and restarts the pod, which then fails again in a restart loop.

**Fix:**  
Add a startup probe with a generous timeout:
```yaml
startupProbe:
  httpGet:
    path: /health/live
    port: 8000
  failureThreshold: 30
  periodSeconds: 10
```
This gives the app 5 minutes to start before liveness kicks in.

### P2-8: Docker image tags pinned to `latest` — non-reproducible deployments

**Location:** `infra/helm/claude-code/values.yaml:7`, `infra/helm/claude-code/Chart.yaml:9`, all `Dockerfile` base images

**What's happening:**

## Patterns & Systemic Issues

### Stub-driven development without a cleanup gate

Multiple features are implemented as stubs with "wire up in later phases" comments: slash commands, `openSettings`, `openSearch`, `_lookup_frame`. The stubs are in the production code path. There's no mechanism (feature flag, assertion, TODO tracker) that prevents these from shipping. A developer could deploy the current code and users would interact with features that silently do nothing.

**Recommendation:** Introduce a convention: stubs raise `NotImplementedError` in development mode and are hidden behind a `FEATURE_FLAG` in the frontend. Stubs that silently succeed are the most dangerous.

### Error swallowing at the edges

Seven independent locations swallow exceptions with `except Exception: pass`. This is a pattern, not a coincidence. It likely reflects a "don't let one failure break the whole system" intent, which is correct — but the implementation is wrong. Silent failures accumulate and make the system appear healthy when it's degrading.

**Recommendation:** Adopt a policy: every `except Exception` block must either re-raise, log with `exc_info=True`, or emit a monitoring event. The `# noqa: BLE001` suppression should be paired with a log call, not a `pass`.

### Infrastructure never tested end-to-end

The Docker and Helm configs have never been tested against the actual backend (evident from the health endpoint mismatch). Infrastructure drift happens incrementally and goes unnoticed.

**Recommendation:** Add an infra smoke test to CI: `docker compose up`, wait for health check, hit `/api/health`, assert 200. This one test would have caught P0-1 and P0-3.
