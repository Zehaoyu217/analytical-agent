## Pre-audit fixes (made during audit)

The first audit run produced **0 candidates**, despite vulture and knip individually flagging many symbols. Three silent failures were diagnosed and repaired before the audit could meaningfully run. None of these were spec defects — all were implementation bugs in the wrappers / matching layer.

### Fix 1 — wrappers couldn't find their binaries from project root

`subprocess.run(["vulture", ...])` fails with `FileNotFoundError` when invoked from the repo root, because `vulture` lives in `backend/.venv/bin/` and is not on `$PATH`. Same for `knip` (`frontend/node_modules/.bin/`). The wrappers caught the error, set `failure_message`, and the rule treated this as "no findings" — silently passing.

**Fix:** wrappers now auto-resolve the project-local binary when given `repo_root`. Failures are surfaced as `graph.dead_code.tool_unavailable` INFO issues so a missing tool can never again silently zero the gate.

Commit: `fix(integrity): resolve vulture/knip from project venv + surface tool unavailability`

### Fix 2 — knip path prefix mismatch

`knip` runs with `cwd=frontend/` and emits paths like `src/lib/api.ts`. The graph stores paths as `frontend/src/lib/api.ts`. The intersection key never matched.

**Fix:** `parse_knip_output()` accepts a `path_prefix` and `run_knip()` computes it from `frontend_dir.relative_to(repo_root)`.

### Fix 3 — graph label vs knip export name format

Graph nodes for functions use labels like `truncate()` (with trailing parens). Knip emits export names as bare identifiers (`truncate`). The label-to-export comparison never matched.

**Fix:** `_strip_call_parens()` helper applied to graph labels before set membership check.

Combined commit: `fix(integrity): normalize knip path prefix + dead_code label parens`

## Notes on confidence and bait

- At default `min_confidence=80`, only 4 candidates surface (all knip-only, the four frontend exports). At `min_confidence=60`, two backend symbols join. Lowering further (50%) would surface unused parameters (`amount`, `exc_type`, `tb`, `attrs`), which the dead_code rule correctly does not match — those are parameter-kind nodes that the graph extractor doesn't track as separate symbols.
- The **0 false-positive rate** is real but deserves a caveat: the triple-intersection is narrow by design. A symbol must be flagged by the language-specific tool **and** be a graph orphan. This means we don't catch symbols that are referenced symbolically (e.g., via dynamic dispatch or `getattr`) but never actually called — that's not a bug, that's the gate working as intended.
- The surviving orphan set is **479 nodes**. Most are React components used through routing/section dispatch that Plugin A's extractor doesn't trace into (e.g., `chatsection`, `agentssection`, etc.). These are NOT flagged because knip correctly sees them as imported (via React's JSX entry points) — the orphan-set false-positive rate is what gate α audited, not gate β.
